# ADR-0001: 从 GenericAgent 抽出 bs-webctl 为独立 PyPI 包

## Context

GenericAgent 是一个更大的 agent monorepo，浏览器控制能力（TMWebDriver +
simphtml）原本嵌在其中。触发本决策的力有三股：

1. **复用摩擦**：其他 AI agent 想用浏览器控制原语，就得拉整个 GenericAgent
   仓库，依赖耦合在 agent 主循环上。
2. **稳定原语 vs 演进主循环**：浏览器控制是相对稳定的能力层，与 agent 主循环
   的演进节奏不同，耦合在一处会让两边都难独立迭代。
3. **生态缺位**：AI agent 生态缺少一个"控制真实 Chrome、保留登录态"的标准
   原语包，每个 agent 都在重造轮子。

## Decision

把 TMWebDriver + simphtml 抽出为独立 PyPI 包 `bs-webctl`，单一 CLI 入口
`bs-webctl`，依赖最小化（`bottle` + `beautifulsoup4` + `requests`），
WebSocket server 内联自 `simple_websocket_server` 纯 stdlib 实现，避免依赖
未维护的 PyPI 包。

## Alternatives Considered

- **选项 A：留在 GenericAgent 内，其他 agent 复制粘贴代码**
  - 优点：零迁移成本
  - 缺点：代码 drift、无版本、bug 修不传播、每个 agent 各自维护一份
  - 为何不选：违背"原语稳定、可复用"诉求，长期成本最高

- **选项 B：抽出但作为 GenericAgent 的 git submodule**
  - 优点：保留单一仓库真相源
  - 缺点：submodule 体验差；`pip install` 无法直接装；消费者仍需 clone
  - 为何不选：不解决"一行安装"诉求，submodule 在 Python 生态里反模式

- **选项 C：抽出为独立 PyPI 包（本决策）**
  - 优点：`pip install bs-webctl` 一行到位；独立版本号；清晰边界
  - 缺点：与 GenericAgent 的双向变更需跨仓库同步
  - 选这个：复用诉求 + 稳定原语诉求 + 生态缺位三者同时满足

## Consequences

- 正面：任何 agent `pip install bs-webctl` 即可用；版本独立可追溯；bug 修一处
  全生态受益；与 agent 主循环解耦，两边独立迭代。
- 负面：浏览器协议层（CLI ↔ 扩展 ↔ WS）的变更需跨仓库协调；GenericAgent 改为
  依赖本包时有一次迁移工作。
- 需后续跟进：
  - 定义协议版本号策略（扩展 ↔ server 握手时协商版本）；
  - GenericAgent 切换为依赖本包时，走 `log.md` 记录迁移决策；
  - WebSocket 内联 stdlib 实现的取舍另立 ADR（后续 ADR-0002 候选）。
